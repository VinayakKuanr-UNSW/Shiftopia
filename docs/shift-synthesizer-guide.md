# Shift Synthesizer: Step-by-Step Operating Guide

This guide provides the instructions for running the ML-driven labor demand forecasting and automated shift generation pipeline.

## 1. Environment Prerequisites

Ensure your `.env` file contains the ML service URL:

```env
VITE_ML_URL=http://localhost:8000
```

## 2. Infrastructure Setup

The Shift Synthesizer requires the ML Python service to be running. The easiest way to start it is via Docker Compose:

```bash
# Start the ML service and model volume mapping
docker compose up -d ml-service
```

Verification: Visit [http://localhost:8000/docs](http://localhost:8000/docs) to ensure the FastAPI server is healthy.

## 3. Data Seeding (VenueOps Events)

To forecast labor, the system needs event data. Use the newly integrated seed generator to populate your local database:

```bash
# 1. Generate the SQL seed file
npm run seed:generate

# 2. Apply the seed to your Supabase project
# You can copy the contents of supabase/seed.sql into the Supabase SQL Editor
```

## 4. Frontend Workflow: Generating Shifts

### Step A: Navigate to Forecasting
1. Open the sidebar and go to **Rostering** > **Labor Demand**.
2. Select your **Department** and **Sub-Department** using the global scope filter.
3. Choose a date range (e.g., next week) to see the forecasted demand curves.

### Step B: Visualize Demand vs. Coverage
*   **Blue Area**: Represents the ML-predicted labor demand (staff counts).
*   **Bars**: Represent your currently rostered shifts.
*   **Gaps**: The UI will highlight periods where demand exceeds coverage.

### Step C: Generate "Dry-Run" Shifts
1. Click the **"Generate Shifts"** button at the top right.
2. The system will contact the ML service, generate staff "tensors", and run a bin-packing algorithm to create proposed shift blocks.
3. Review the **Preview Modal** showing the exact shifts the system intends to create.

### Step D: Inject into Roster
1. If satisfied with the preview, click **"Inject Shifts"**.
2. These shifts will be created in a **Draft** state and assigned a `synthesis_run_id`.
3. They will immediately appear on your Roster Grid.

### Step E: Understanding Generated Shifts (XAI)
The shift synthesizer uses Explainable AI (XAI) to ensure transparency:
*   **Proposed Adjustments Panel**: In the Labor Demand page, hover over the **"Why?" (Sparkles)** badge next to proposed additions to see the exact events driving the labor requirement.
*   **Shift Cards**: On the Roster Grid, autogenerated shifts display a specialized **XAI Badge** (or a Sparkles icon in the compact view). Hovering over this indicator will reveal the "Generated to meet demand for..." reasoning, explicitly listing the events that triggered the shift's creation.

## 5. Audit & Rollback

If the generation was incorrect or needs adjustment:
1. Stay on the **Labor Demand** page.
2. Look at the **"Recent Synthesis Runs"** table at the bottom.
3. Find your last run and click **"Rollback"**.
4. This will perform a cascaded deletion of all shifts created during that specific session.

---

### Troubleshooting
*   **ML Service Error**: Ensure the Docker container is running and port 8000 is not blocked.
*   **No Demand Data**: Verify that `venueops_events` and `venueops_functions` have data for the selected date range.
*   **Role Mapping Issues**: Check the `role_ml_class_map` table in the database to ensure your internal roles are mapped to ML classes (e.g., 'Usher', 'Security').
