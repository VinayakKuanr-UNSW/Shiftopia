[DONE 2026-04-29] All 11 ERROR-level advisor findings resolved
  - 5 unprotected `public` tables → RLS enabled.
    Migration: 20260429_enable_rls_on_5_unprotected_tables.sql
  - 6 SECURITY DEFINER views → SECURITY INVOKER.
    Migration: 20260429_convert_views_to_security_invoker.sql

[DONE 2026-04-29] 187 SECURITY DEFINER functions callable by `anon`
  - Root cause was the implicit PUBLIC grant, not a direct anon grant.
    REVOKE EXECUTE ... FROM PUBLIC closes it; authenticated/service_role
    keep their explicit grants and continue working.
  - Migration: 20260429_revoke_anon_execute_on_definer_funcs.sql

[DONE 2026-04-29] 271 functions without SET search_path
  - All SET to `pg_catalog, public`. Preserves existing unqualified
    references while pinning resolution order (closes injection vector).
  - Migration: 20260429_harden_function_search_paths.sql

[DONE 2026-04-29] Materialized view `employee_daily_metrics` exposed via API
  - REVOKED SELECT from anon, authenticated. service_role keeps access
    so refresh RPCs continue to work. App had zero direct queries
    against the view (only schema-graph reference in TS types).
  - Migration: 20260429_lockdown_employee_daily_metrics_matview.sql

[DONE 2026-04-29] `broadcast-attachments` storage bucket lockdown
  - Dropped overly broad authenticated SELECT policy. Public bucket
    URLs (getPublicUrl) still work; only bucket-wide enumeration via
    storage.list() is closed.
  - Migration: 20260428_lockdown_broadcast_attachments_bucket.sql

[PARTIAL] Schema drift between live DB and supabase/migrations/
  - Live DB has 400+ legacy migrations recorded in history that aren't
    in the repo. The 8 migrations added during this hardening pass ARE
    in both places (repo + live history table) and are reviewable in
    PR diffs.
  - Real fix to close the legacy gap: run `migration repair --status
    reverted` for all 400 historic entries, then `db pull` to squash
    into a single baseline. ~1h focused work.
  - Until done: continue Option A workflow — apply new DDL via MCP
    (or Studio), commit the matching .sql file.

[TODO] 74 RLS policies with USING (true) / WITH CHECK (true)
  - 46 distinct tables affected, including timesheets, swap_requests,
    organizations, departments, pay_periods, work_rules, system_config.
  - Each requires a BUSINESS-RULE decision (employee-owns-row?
    manager-owns-dept? org-scoped?). Cannot be done as a bulk pass.
  - Highest-effort and highest-impact item remaining. This is the
    actual multi-tenant data-leak surface.
  - Recommended approach: one PR per table, starting with `timesheets`
    and `swap_requests` (most sensitive). Use existing helper functions
    (is_admin, has_permission, user_has_action_in_scope) as building
    blocks. Many policies in `events` (now done) follow a reusable
    pattern: `is_admin() OR <fk> IN (SELECT ... WHERE user_id = auth)`.

[TODO] 187 `authenticated_security_definer_function_executable` warnings
  - These are the same 187 functions, now flagged because authenticated
    users CAN call them. By design — that's what business RPCs are for.
  - Clearing this lint requires rewriting each function as SECURITY
    INVOKER plus matching RLS policies on every table they touch. A
    multi-week project with dubious ROI vs. the USING(true) work above.
  - Recommend treating this lint as ACCEPTED INTENT and marking it
    suppressed in advisor config (when Supabase exposes that config).

[BLOCKED] HaveIBeenPwned password check
  - Supabase Auth feature — only available on Pro plan and above.
  - Re-evaluate when upgrading. Until then, document in user-facing
